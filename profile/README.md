# The Community

**Team Killing Bastards — TKB — is a Scottish-run gaming community and technical project home.**

The community was founded and originally created by **[Conroy1988](https://github.com/Conroy1988)**. It is led by Conroy alongside **[MartyBlyth](https://github.com/Martyblyth)**, his right-hand and fellow community leader.

This GitHub organisation exists to support the community with software, automation, game tooling and controlled technical infrastructure. It is not a dumping ground for unrelated repositories: each maintained system has a defined purpose, accountable owner, canonical source, deployment path and security boundary.

## Operating model

| Principle | What it means inside TKB |
|---|---|
| **Community before tooling** | Technology exists to support real community activity, administration and gameplay. |
| **Clear technical authority** | Each system has an identified maintainer with explicit development and release responsibility. |
| **Separate trust domains** | Public software and private community infrastructure do not share runtime, credentials or authority. |
| **Evidence over assumption** | Releases, deployments, backups and operational changes are validated against the live result. |
| **Maintenance is part of delivery** | Documentation, issue tracking, recovery, migration and repository governance are treated as product work. |

### Secure community automation and operational control platform

The **TKB Discord Bot** is the private operational backbone of the Team Killing Bastards Discord environment. It combines a live Discord runtime, secure administration dashboard, transactional persistence and restricted deployment-management system as one controlled service.

**Platform capability**

- Discord commands, events, scheduled tasks and community automation

- Battlefield profiles, levels, ranks, leaderboards and title cards
- Starboard, moderation logging, messaging and countdown management
- Docker deployment with optional isolated Caddy HTTPS termination
- Sanitised telemetry, validated backups, controlled updates and automatic rollback

# Delivery and Engineering Standard

<table>
<tr>
<td width="25%" align="center" valign="top">

### ◈ Source Authority

A canonical repository and trusted branch for every maintained system.

</td>
<td width="25%" align="center" valign="top">

### ✓ Verified Change

Review, validation and release controls appropriate to each project.

</td>
<td width="25%" align="center" valign="top">

### ⛨ Security Boundary

Secrets, runtime data and private operations remain outside public source.

</td>
<td width="25%" align="center" valign="top">

### ↗ Traceable Delivery

Documented versions, deployment paths, recovery and accountable ownership.

</td>
</tr>
</table>

TKB repositories are expected to maintain:

- A precise project purpose and clearly defined scope
- Accurate documentation aligned with the live system
- Explicit maintainer authority and supporting responsibilities
- Reproducible deployment or installation instructions
- Security controls proportionate to exposure and operational risk
- Structured issue, change and release workflows where applicable
- Validated release artefacts and recoverable delivery processes
- Clear separation between unrelated products and technical domains

## Command Nexus release path

```text
Focused source change
        ↓
Version and changelog update
        ↓
Pull request validation and review
        ↓
Merge into trusted main
        ↓
Automatic unpublished-version detection
        ↓
Recoverable GitHub Release publication
        ↓
Asset download and checksum verification
        ↓
Immutable GitHub source and Greasy Fork parity
        ↓
Discord release announcement
```

## TKB Discord platform trust model

```text
Discord Guild ←→ Discord Runtime
                     ↕
             Transactional SQLite
                     ↕
Authenticated Control Centre ←→ Sanitised Operations Bridge
                                      ↕
                          Restricted Windows Host Agent
                                      ↕
                         Git · Docker · Backups · Recovery
```

The public-facing application receives no Docker socket and exposes no generic shell, arbitrary command, path or script execution endpoint. Controlled update, backup and restore operations cross a deliberately restricted host boundary.
