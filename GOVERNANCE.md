# Governance

## Organisation ownership

Team Killing Bastards is administered by its GitHub organisation owners.

Organisation ownership provides administrative continuity; it does not imply equal technical authorship of every project.

## Project authority

Each repository should state:

- Project creator or lead developer.
- Technical decision-maker.
- Release approver.
- Supporting administrative or documentation roles.

For **MissionChief Command Nexus**:

- **MartyBlyth** is the project creator, userscript developer, technical decision-maker and final release approver.
- **Conroy1988** is a project helper for repository setup, documentation and general support. He is not a userscript developer.

## Technical decisions

Technical decisions should prioritise:

1. User safety and data preservation.
2. Correct operational behaviour.
3. Maintainability and controlled lifecycle management.
4. Compatibility supported by evidence.
5. Clear migration and release traceability.

Substantial architectural changes should be discussed in an issue before implementation.

## Pull requests

Repositories may require pull requests, review and automated checks before changes reach the default branch.

Code ownership and review rules may distinguish between:

- Source code and release automation.
- Documentation and repository presentation.
- Community templates and administrative files.

## Releases

A release should not be treated as approved merely because packaging or documentation is complete. Project-specific technical approval remains with the stated developer or release authority.

## Disputes

Where contributors disagree:

1. Record the technical question clearly.
2. Present evidence, reproduction steps or trade-offs.
3. Separate factual disagreement from preference.
4. Defer to the project's stated technical authority after reasonable review.

Organisation owners retain final authority over organisation security, access and legal risk.
