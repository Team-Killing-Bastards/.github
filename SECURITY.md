# Security Policy

## Supported versions

Security support is normally limited to the latest published release and any explicitly identified pre-release currently under active testing.

Repositories that have not yet published a release should be treated as development projects rather than production-ready software.

## Reporting a vulnerability

Do not publish exploitable details, account information, tokens, private URLs or personal data in a public issue.

Use the target repository's **Security** tab and private vulnerability-reporting facility when available. If private reporting is not available, contact an organisation owner privately before opening a public issue.

Include:

- Affected project and version.
- Impact and realistic exploitation conditions.
- Reproduction steps or proof of concept.
- Relevant browser and userscript-manager details.
- Suggested mitigation, where known.

## Response process

A report may be:

1. Acknowledged and reproduced.
2. Assessed for severity and affected versions.
3. Fixed privately where disclosure would create risk.
4. Released with appropriate migration or mitigation guidance.
5. Disclosed publicly after users have a reasonable opportunity to update.

## Scope

Security reports may include:

- Exposure of authentication material or private data.
- Unsafe cross-origin behaviour.
- Unintended destructive actions.
- Repeat dispatch, rename or personnel-assignment loops.
- Storage corruption or unsafe migration.
- Supply-chain or release-artefact tampering.

Normal gameplay defects, feature requests and support questions should use the repository's standard issue forms instead.
