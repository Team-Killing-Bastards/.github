# Contributing to Team Killing Bastards Projects

Thank you for taking the time to improve a Team Killing Bastards project.

## Before contributing

1. Read the target repository's README, roadmap and existing issues.
2. Search for an existing issue before opening a duplicate.
3. Discuss substantial architectural or behavioural changes before implementing them.
4. Keep each pull request focused on one coherent change.

## Bug reports

A useful bug report includes:

- Project and version.
- MissionChief domain.
- Browser and version.
- Userscript manager and version.
- Clear reproduction steps.
- Expected and actual behaviour.
- Relevant console output or screenshots with private information removed.
- Other userscripts active during reproduction.

## Feature proposals

Explain the operational problem first. Include:

- Current workflow.
- Proposed behaviour.
- Who benefits.
- Safety, compatibility and migration implications.
- Any alternatives already considered.

## Pull requests

Pull requests should:

- Reference the relevant issue.
- Explain what changed and why.
- Avoid unrelated formatting or refactoring.
- Preserve existing user data unless migration is explicitly documented.
- Include validation evidence.
- Update documentation and changelogs where user-visible behaviour changes.
